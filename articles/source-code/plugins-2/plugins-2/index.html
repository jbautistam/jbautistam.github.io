Desarrollo de plugins con MEF

Oct 11, 2019 Programación
Share on:

Hace ya algunos años, escribía sobre la creación de plugins para aplicaciones.NET utilizando la primera versión de MEF.

Por recordarlo, un plugin es una aplicación o librería a la que podemos acceder desde nuestra aplicación sin necesidad de enlazarla en tiempo de compilación. Es decir, no tenemos que añadir una referencia en nuestro proyecto, al ejecutar nuestra aplicación se carga la dependencia externa y se realizan las llamadas oportunas.

A muchos les sonará como inyección de dependencias y en cierto modo es así, la única diferencia real es que en ninguna parte de nuestro proyecto o solución necesitamos una referencia a la librería que implementa las interfaces.

En esta ocasión vamos a utilizar la última versión de MEF (en algún momento se llamó MEF 2 pero finalmente Microsoft ha optado por eliminar el número de versión).

Ha habido algún cambio importante respecto a la primera versión que describía en el artículo anterior, quizá el más importante es que se ha eliminado la necesidad de incluir atributos [Export] en los plugins indicando que cumplían con un interface. Además ha desaparecido el método para cargar ensamblados de un directorio y ahora hay que añadirlos manualmente al contenedor, pero en general, se ha mejorado bastante la forma de trabajar con los componentes.

El interface de los plugins en el host

Para trabajar con plugins, vamos a utilizar dos proyectos, el primero de ellos, para entendernos, lo vamos a llamar host: este proyecto es el ejecutable o ensamblado principal de nuestra aplicación y será el encargado de lanzar los plugins.

Para comenzar, debemos incluir en nuestro proyecto de Host el paquete de Nuget System.Composition que es el que nos ofrece las funcionalidades de MEF.

En segundo lugar, necesitamos un interface que es el que deben implementar los plugins. Por supuesto, el interface lo podemos definir como cualquier otro interface. Para el ejemplo, he decidido crear una consola que lance procesadores de comandos o pasos de trabajo, por eso he definido el siguiente interface para los plugins:

 1using System;
 2using System.Threading.Tasks;
 3
 4using Bau.Libraries.LibJob.Application.Models.Processes;
 5
 6namespace Bau.Libraries.LibJob.Application.Interfaces
 7{
 8	public interface IJobStepProcessor
 9	{
10		void Initialize(Models.Context.JobContextModel context, ProcessStepModel step);
11
12		Task<bool> ProcessAsync();
13
14		string Key { get; }
15	}
16}

Lo que pretendo es que haya diferentes procesadores de comandos a los que se les envíe la información de contexto (donde se podrían incluir cadenas de conexión, directorios, nombres de archivos...) y los datos del paso (dónde estaría por ejemplo el nombre del archivo de script y sus parámetros, por ejemplo).

Por supuesto, el plugin también debe tener acceso a las definiciones de estos objetos que estarían en una librería de módulos comunes. Por no hacer más largo el ejemplo, no vamos a ver estas definiciones pero las podéis encontrar en el código fuente en GitHub.

El proyecto de plugin

Nuestro proyecto de plugin, para este caso, el procesador de pasos, debe incluir también el paquete System.Composition e implementar la interface que hemos definido en el host.

Para que sea más sencillo, suelo definir en la librería común donde se encuentra la interface de los plugins una clase base abstracta encargada de la 'fontanería' de logging, propiedades y demás aunque no es estrictamente necesario:

 1using System;
 2using System.Threading.Tasks;
 3
 4using Bau.Libraries.LibJob.Application.Models.Processes;
 5
 6namespace Bau.Libraries.LibJob.Application.Processor
 7{
 8	public abstract class JobStepProcessorBase: Interfaces.IJobStepProcessor
 9	{
10		public JobStepProcessorBase(string key)
11		{
12			Key = key;
13		}
14
15		public void Initialize(Models.Context.JobContextModel context, ProcessStepModel step)
16		{
17			Context = context;
18			Step = step;
19		}
20
21		public async Task<bool> ProcessAsync()
22		{
23			Context.UpdateJobStep(Key, Step);
24			Context.WriteDebug("Start process");
25			await ExecuteStepAsync();
26			Context.WriteDebug("End process");
27			return Context.Errors.Count == 0;
28		}
29		
30		protected abstract Task ExecuteStepAsync();
31
32		public string Key { get; }
33
34		public ProcessStepModel Step { get; private set; }
35
36		protected Models.Context.JobContextModel Context { get; private set; }
37	}
38}

Y en la librería de plugins, nos quedaría por tanto heredar de la clase abstracta e implementar el método ExecuteStepAsync.

Para que el ejemplo sea sencillo: el plugin no ejecuta ningún proceso 'práctico', simplemente lee un archivo de texto donde en teoría están las instrucciones del paso, sustituye ciertos marcadores en el texto por los argumentos pasados desde el host y lo muestra en la consola:

 1using System;
 2using System.Threading.Tasks;
 3
 4using Bau.Libraries.LibCommonHelper.Extensors;
 5
 6namespace Bau.Libraries.LibJob.Processor.Test
 7{
 8	public class TestProcessor: Application.Processor.JobStepProcessorBase
 9	{
10		public TestProcessor(): base("TestProcessor") {}
11
12		protected async override Task ExecuteStepAsync()
13		{
14			if (!System.IO.File.Exists(Step.ScriptFileName))
15				Context.WriteError($"Can't find the file {Step.ScriptFileName}");
16			else
17			{
18				string text = System.IO.File.ReadAllText(Step.ScriptFileName);
19
20					text = ReplaceParameters(text);
21					Context.WriteInfo(text);
22			}
23			await Task.Delay(100);
24		}
25
26		private string ReplaceParameters(string text)
27		{
28			foreach ((string key, object value) in Step.GetCombinedParameters().Enumerate())
29				if (value == null)
30					text = text.ReplaceWithStringComparison("{{" + key + "}}", "NULL");
31				else
32					text = text.ReplaceWithStringComparison("{{" + key + "}}", value.ToString());
33			return text;
34		}
35	}
36}

La conexión del host con el plugin

Hasta ahora, tenemos dos librerías separadas, una de host y otra de plugin, pero no hemos conectado ambas aún. De alguna forma debemos indicar en el host que se carguen los plugins en el contenedor de MEF.

Hay varias formas de hacerlo que se pueden ver en la documentación. Para mí, la más sencilla es generar los plugins en directorios y añadir al contenedor los ensamblados de esos directorios que cumplen con la interface. Veamos el código de carga de plugins:

 1using System;
 2using System.Collections.Generic;
 3using System.Composition;
 4using System.Composition.Hosting;
 5using System.Linq;
 6using System.Reflection;
 7
 8namespace Bau.Libraries.LibJob.Application.Processor.Controllers
 9{
10	internal class PluginsManager<TypePlugin>: IDisposable
11	{
12		public void Initialize(List<string> pathPlugins, string extensionPlugins = ".plugin.dll")
13		{
14			Container = new ContainerConfiguration().WithAssemblies(GetPluginsAssemblies(pathPlugins, NormalizeExtension(extensionPlugins)), 
15																	GetConventionsBuilder()).CreateContainer();
16			Plugins = Container.GetExports<TypePlugin>().ToList();
17		}
18
19		private string NormalizeExtension(string extension)
20		{
21			if (string.IsNullOrWhiteSpace(extension))
22				extension = ".plugin.dll";
23			extension = extension.Trim();
24			if (!extension.StartsWith("."))
25				extension = "." + extension;
26			return extension;
27		}
28
29		private IEnumerable<Assembly> GetPluginsAssemblies(List<string> pathPlugins, string extensionPlugins)
30		{
31			List<Assembly> assemblies = new List<Assembly>();
32
33				foreach (string path in pathPlugins)
34					if (System.IO.Directory.Exists(path))
35						foreach (string fileName in System.IO.Directory.GetFiles(path, "*" + extensionPlugins))
36							if (System.IO.File.Exists(fileName) && fileName.EndsWith(extensionPlugins, StringComparison.CurrentCultureIgnoreCase))
37								try
38								{
39									assemblies.Add(Assembly.LoadFrom(fileName));
40								}
41								catch (Exception exception)
42								{
43									Errors.Add($"Error when load plugin {fileName}{Environment.NewLine}{exception.Message}");
44								}
45				return assemblies;
46		}
47
48		private System.Composition.Convention.ConventionBuilder GetConventionsBuilder()
49		{
50			System.Composition.Convention.ConventionBuilder builder = new System.Composition.Convention.ConventionBuilder();
51
52				builder.ForTypesDerivedFrom<TypePlugin>().Export<TypePlugin>().Shared();
53				return builder;
54		}
55
56		protected virtual void Dispose(bool disposing)
57		{
58			if (!Disposed)
59			{
60				if (disposing)
61					Container = null;
62				Disposed = true;
63			}
64		}
65
66		public void Dispose()
67		{
68			Dispose(true);
69		}
70
71		[ImportMany]
72		public IEnumerable<TypePlugin> Plugins { get; private set; }
73
74		private CompositionContext Container { get; set; }
75
76		public List<string> Errors { get; } = new List<string>();
77
78		public bool Disposed { get; private set; }
79	}
80}

Sí, un poco largo, lo sé, pero en realidad lo importante es que la clase recibe una lista de directorios donde en teoría se encuentran los plugins y los añade a un contenedor utilizando la siguiente instrucción:

1Container = new ContainerConfiguration().WithAssemblies(GetPluginsAssemblies(pathPlugins, NormalizeExtension(extensionPlugins)), 
2					GetConventionsBuilder()).CreateContainer();

El método GetPluginsAssemblies lee los ensamblados de los diferentes directorios y los añade a una lista de tipo Assembly. La variable extensionPlugins identifica la extensión de los archivos de plugins, puede ser simplemente .dll pero en caso de proyectos con muchos ensamblados, quizá sea mejor marcar el ensamblado principal (el que cumple con la interface de plugin) con una extensión particular como por ejemplo .plugin.dll (de todas formas, esto no es obligatorio).

Por su parte, el método GetConventionsBuilder() se asegura que sólo cargamos en el contenedor los ensamblados que cumplen con la interface definida en el genérico TypePlugin.

Nota: Para los puristas, en nuestro caso, no implementa directamente la interface si no que hereda de una clase abstracta que implementa la interface. Funciona igualmente.

Cuando finaliza la carga, en la propiedad Plugins vamos a tener cargados todos los plugins que hemos obtenido del contenedor con esta instrucción del método Initialize:

1Plugins = Container.GetExports<TypePlugin>().ToList();

Ejecución de los plugins

Por último, lo que nos queda es aprovechar los plugins que tenemos cargados y ejecutarlos.

En el caso de ejemplo, nuestra aplicación principal, carga los plugins y los inserta en una lista de procesadores:

 1Processor.Controllers.PluginsManager<Processor.JobStepProcessorBase> pluginsManager = new Processor.Controllers.PluginsManager<Processor.JobStepProcessorBase>();
 2
 3pluginsManager.Initialize(configuration.PathPlugins, ".dll");
 4if (pluginsManager.Errors.Count == 0)
 5{
 6	foreach (Processor.JobStepProcessorBase processor in pluginsManager.Plugins)
 7		Processors.Add(processor.Key, processor);
 8	initialized = true;
 9}
10else
11	foreach (string pluginError in pluginsManager.Errors)
12		Context.Logger.WriteError("JobManager - Initialize plugins", pluginError);

Y llama al procesador adecuado para la ejecución de cada paso:

 1private async Task ProcessStepAsync(ProcessStepModel step)
 2{
 3	Processor.JobStepProcessorBase processor = Processors[step.PluginKey];
 4
 5		if (processor == null)
 6			Context.WriteError("JobManager - Process", $"Can't find the processor for step {step.PluginKey}");
 7		else
 8		{
 9			try
10			{
11				processor.Initialize(Context, step);
12				await processor.ProcessAsync();
13			}
14			catch (Exception exception)
15			{
16				Context.WriteError("JobManager - Process", exception.Message);
17			}
18		}
19}

Una nota importante que me dio algunos problemas al principio: si tenemos el código fuente del host y los plugins en una solución de Visual Studio, lo podemos ejecutar y depurar sin problemas pero cuando hagamos modificaciones debemos recompilar toda la solución porque los plugins no se compilan siempre cuando depuramos.

Supongo que es porque la optimización de la compilación busca las referencias de proyectos en la librería que contiene el ejecutable pero como en realidad no hay ninguna referencia a los plugins, no los compila de nuevo. Por eso hay que hacerlo a mano o modificar la configuración del compilador para que siempre compile toda la solución. Yo acabé asociando la tecla F7 al comando 'Recompilar toda la solución' pero va en gustos.

Y eso es todo, simple y limpio.

Como siempre, podéis ver el código fuente en GitHub.
