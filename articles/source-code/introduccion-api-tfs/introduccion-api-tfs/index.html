Introducción a la API de TFS

Oct 11, 2019 Programación
Share on:

Estos días he estado jugando con el acceso a Team Foundation Server utilizando sus API y servicios Web.

Aunque una vez que conoces sus bases es bastante fácil de utilizar, la documentación es, en el mejor de los casos, escasa y dispersa por eso he decidido escribir una breve introducción a TFS para programadores.

Por supuesto, antes de comenzar debemos utilizar NuGet para descargar y añadir las referencia al API. En mi caso he añadido las referencias a:

  • Microsoft.TeamFoundationServer.Client
  • Microsoft.TeamFoundationServer.ExtendedClient

Aunque las APIs de Team Foundation Server nos ofrecen acceso en teoría a todos los módulos de TFS, a mí me interesaba particularmente el acceso a los proyectos y tareas, por eso este artículo está relacionado con estos elementos.

Conexión a TFS

En primer lugar, necesitamos definir el objeto que nos va a dar el acceso a los diferentes apartados de la API, he definido esta variable global en la clase que almacena mis métodos de acceso a TFS:

1private TfsTeamProjectCollection tfsTeamProject = null;

Lo primero que debemos hacer por supuesto es conectarnos con el servidor de Team Foundation Server, para ello utilizo este código:

 1public void Connect(string strUrl, string strLogin, string strPassword)
 2{ 
 3	WindowsCredential objWindowsCredentials = new WindowsCredential(new NetworkCredential(strLogin, strPasword));
 4  	TfsClientCredentials objCredentials = new TfsClientCredentials(objWindowsCredentials);
 5
 6	// Crea una conexión
 7	tfsTeamProject = new TfsTeamProjectCollection(strUrl, objCredentials);
 8	// Autentifica
 9	tfsTeamProject.Authenticate();
10}

La URL del servicio de Team Foundation Server se compone de la URL del servidor TFS al que estamos conectando y la colección a la que queremos conectar. En mis experimentos he utilizado la versión online de TFS, es decir, la cuenta gratuita que obtuve en la web de Visual Studio.

Esta Web me da una Url de acceso del tipo https://usuario.visualstudio.com. Como colección utilizo la predeterminada DefaultCollection. Así, la Url que le pasamos al método debe ser en este caso: https://usuario.visualstudio.com/DefaultCollection.

Por supuesto, el login y la contraseña son los que nos ha dado el sistema inicialmente.

Proyectos en TFS

Para cargar los proyectos que tenemos en TFS, una vez abierta la conexión utilizo un método similar a este:

1WorkItemStore objWorkItemStore = tfsTeamProject.GetService<WorkItemStore>();
2
3// Carga los proyectos
4foreach (Project objTfsProject in objWorkItemStore.Projects)
5 .... tratamiento del proyecto de TFS

Como se puede ver en el código fuente en GitHub, he definido una serie de clases de modelo (como ProjectModel) que mantienen los datos en memoria para no tener que acceder continuamente al servidor. Estas clases de modelo son bastante sencillas. Es decir: propiedades con el nombre, la descripción, el Id y colecciones de elementos relacionados (en el caso de los proyectos tendríamos por ejemplo la colección de tareas):

 1/// < summary>
 2///	Obtiene los proyectos
 3/// < /summary>
 4private ProjectModelCollection GetProjects()
 5{ 
 6	ProjectModelCollection objColProjects = new ProjectModelCollection();
 7  	WorkItemStore objWorkItemStore = tfsTeamProject.GetService<WorkItemStore>();
 8
 9	// Carga los proyectos
10	foreach (Project objTfsProject in objWorkItemStore.Projects)
11	{ 
12		ProjectModel objNewProject = new ProjectModel();
13		
14		// Asigna los datos
15		objNewProject.ID = objTfsProject.Id;
16		objNewProject.Name = objTfsProject.Name;
17		// Asigna las categorías
18		objNewProject.Categories.AddRange(LoadCategories(objTfsProject));
19		// Añade los tipos de tareas
20		for (int intIndex = 0; intIndex < objTfsProject.WorkItemTypes.Count; intIndex++)
21			objNewProject.TaskTypes.Add(objTfsProject.WorkItemTypes[intIndex].Name);
22		// Recorre las iteraciones
23		for (int intIndex = 0; intIndex < objTfsProject.IterationRootNodes.Count; intIndex++)
24		{ 
25			NodeModel objNode;
26			IterationModel objIteration = new IterationModel();
27			
28			// Añade el nodo
29			objNode = objNewProject.Iterations.Add(null, objTfsProject.IterationRootNodes[intIndex].Id,
30			                               objTfsProject.IterationRootNodes[intIndex].Name, objIteration);
31			// Asigna los objetos Lazy
32			objIteration.LazyTasks.LazyData = new Lazy<TaskModelCollection>(() => LoadTasksIteration(objNewProject, objNode.ID?? 0));
33		}
34		// Añade los objetos Lazy
35		objNewProject.LazyTasks.LazyData = new Lazy<TaskModelCollection>(() => LoadTasks(objNewProject));
36		// Y añade el proyecto a la colección
37		objColProjects.Add(objNewProject);
38	}
39	// Devuelve los proyectos
40	return objColProjects;
41}

Si nos fijamos, la clase de proyecto de TFS (Project) nos da acceso no solo a los datos del proyecto si no también a sus categorías, tipos de tarea e iteraciones (entre otros).

Para cargar las tareas, utilizo un objeto de tipo Lazy es decir, no se cargarán las tareas hasta que no se vayan a mostrar, esto permite mejorar el rendimiento de la aplicación puesto que no hacemos la llamada al servidor hasta que no es realmente necesario.

Tareas en TFS

La carga de tareas en TFS es ligeramente más truculenta porque se realiza a partir de consultas, no existe un método del proyecto que nos descargue todas las tareas de un proyecto si no que tenemos que realizar la consulta adecuada. Existen formas para almacenar consultas a partir de objetos del mismo modo que lo hace Visual Studio pero para mi aplicación he utilizado el acceso por consultas SQL en formato de texto.

Como ejemplo, aquí dejo el código para cargar todas las tareas de un proyecto y de una iteración:

 1/// < summary>
 2///	Obtiene los WorkItems
 3/// < /summary>
 4private TaskModelCollection LoadTasks(ProjectModel objProject)
 5{ 
 6	return QueryWorkItems(objProject, $"SELECT * FROM WorkItems WHERE [System.TeamProject] = '{objProject.Name}'");
 7}
 8
 9/// < summary>
10///	Carga las tareas de una iteración
11/// < /summary>
12private TaskModelCollection LoadTasksIteration(ProjectModel objProject, int intIDIteration)
13{ 
14	return QueryWorkItems(objProject, $"SELECT * FROM WorkItems WHERE [System.TeamProject] = '{objProject.Name}' AND [System.IterationId] = '{intIDIteration}'");
15}
16
17/// < summary>
18///	Consulta los elementos de trabajo y los transforma en una colección de tareas
19/// < /summary>
20private TaskModelCollection QueryWorkItems(ProjectModel objProject, string strQuery)
21{ 
22	Project objTfsProject = LoadProjectTfs(objProject);
23	TaskModelCollection objColTasks = new TaskModelCollection();
24	
25	// Carga las tareas del proyecto
26	if (objTfsProject != null)
27	{ 
28		WorkItemStore objWorkItemStore = tfsTeamProject.GetService<WorkItemStore>();
29		WorkItemCollection objColWorkItems = objWorkItemStore.Query(strQuery);
30		
31		// Transforma los elementos de trabajo en tareas
32		foreach (WorkItem objWorkItem in objColWorkItems)
33		{ 
34			TaskModel objTask = new TaskModel();
35			
36			  // Asigna los datos
37			  objTask.ID = objWorkItem.Id;
38			  objTask.Name = objWorkItem.Title;
39			  objTask.Description = objWorkItem.Description;
40			  objTask.IdIteration = objWorkItem.IterationId;
41			  objTask.Project = objWorkItem.Project.Name;
42			  objTask.AssignedTo = GetWorkItemField(objWorkItem, CoreField.AssignedTo);
43			  objTask.DateUpdate = objWorkItem.ChangedDate;
44			  objTask.UserUpdate = objWorkItem.ChangedBy;
45			  objTask.DateNew = objWorkItem.CreatedDate;
46			  objTask.UserNew = objWorkItem.CreatedBy;
47			  objTask.State = objWorkItem.State;
48			  objTask.Tags = objWorkItem.Tags;
49			  objTask.Type = objWorkItem.Type.ToString();
50			  objTask.DateAuthorized = objWorkItem.AuthorizedDate;
51			  objTask.UserAuthorized = GetWorkItemField(objWorkItem, CoreField.AuthorizedAs);
52			  objTask.BoardColumn = GetWorkItemField(objWorkItem, CoreField.BoardColumn);
53			  objTask.BoardColumnDone = GetWorkItemField(objWorkItem, CoreField.BoardColumnDone);
54			  objTask.BoardLane = GetWorkItemField(objWorkItem, CoreField.BoardLane);
55			  objTask.LinkType =  GetWorkItemField(objWorkItem, CoreField.LinkType);
56			  objTask.NodeName = objWorkItem.NodeName;
57			  objTask.Reason = objWorkItem.Reason;
58			  objTask.RelatedLinkCount = GetWorkItemField(objWorkItem, CoreField.RelatedLinkCount).GetInt(0);
59			  // Añade la tarea a la colección
60			  objColTasks.Add(objTask);
61		}
62	}
63	// Devuelve la colección
64	return objColTasks;
65}
66
67/// < summary>
68///	Obtiene un campo de una consulta de elementos de trabajo
69/// < /summary>
70private string GetWorkItemField(WorkItem objWorkItem, CoreField intIDField)
71{ 
72	object objValue = null;
73	
74	// Obtiene el valor
75	if (objWorkItem.Fields.Contains((int) intIDField))
76		objValue = objWorkItem.Fields[intIDField].Value;
77	// Convierte el valor a una cadena
78	if (objValue == null)
79		return null;
80	else
81		return objValue.ToString();
82}

Debemos tener en cuenta que la carga de tareas nos da los WorkItem en una lista, es decir, sabemos que los WorkItem pueden tener tareas asociadas pero TFS no las almacena ni las devuelve como un árbol si no que nos devuelve una lista de relaciones para cada tarea en la colección Links. En en esta colección donde nos indica las relaciones entre las tareas, es decir, si una tarea es hija de o padre de o se tiene que ejecutar antes o después de.
